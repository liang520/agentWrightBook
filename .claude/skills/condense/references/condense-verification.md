# Phase 3 终验 — 9 维度检查

## 前置条件

meta.json `status` 为 `"verifying"`

## 9 维度检查

**维度 1：总字数**
```
统计全部精炼后章节的总字数（统一口径）
通过条件：target_total × 0.95 ≤ 总字数 ≤ target_total × 1.05
```

**维度 2：伏笔完整性**
```
读取 condense-plan.md 中的校准后伏笔状态表
逐条在精炼后的章节中搜索确认：
  - 每个"已埋设"的伏笔在对应章节中仍有体现
  - 每个"已回收"的伏笔在对应章节中仍有体现
标记任何遗漏
```

**维度 3：情绪曲线**
```
从 condense-plan.md 的事件清单中提取情绪高点章节（关键转折、高潮对决）
逐一阅读这些章节的精炼稿：
  - 情绪力度是否因精炼而减弱？
  - 是否存在"铺垫被删导致高潮来得突兀"的问题？
  - 连续章节之间的情绪过渡是否自然？
```

**维度 4：A/B 对照抽检**
```
随机抽取 3-5 章（覆盖不同等级）
将原稿和精炼稿并排对比：
  - 精炼稿是否丢失了原稿中让人印象深刻的细节？
  - 精炼稿的句式是否变得过于单调（所有句子都变短了）？
  - 精炼稿是否保持了原稿的节奏感？
```

**维度 5：因果链完整性**
```
从 condense-plan.md 事件清单中提取跨章因果链：
  - 章 A 的事件 X → 导致章 B 的事件 Y
逐条验证因果链两端是否都在精炼稿中保留
```

**维度 6：跨章状态一致性**
```
提取全部精炼后章节中【】包裹的系统提示文本
筛选含"完成/未完成/触发/失败/达成/进行中"等状态词的提示
按事件名（支线名/任务名）分组
检查同一事件在不同章节是否出现矛盾状态（如先"完成"后"未完成"）
排除合理的状态推进（如"触发"→"进行中"→"完成"是正常流转）
```

**维度 7：数值链闭合**
```
提取全书出现的货币/积分/数值系统名称
在全部章节中搜索含该数值名的行
提取数值和加减操作，验证跨章余额是否闭合
允许"隐性扣除"（如手续费）但必须有文本交代
如全书无数值系统 → 跳过此维度，标记 N/A
```

**维度 8：原作名子串扫描**
```
如果原书有 character-map.md（即原书是 /novel 或 /short 产出）：
  - 对 character-map.md 左列每个原作角色名，提取≥2字的名字部分（去姓氏）
  - 用该子串扫描全部精炼后的章节
  - 白名单过滤：跳过该子串作为常见词的用法（如"依然"作为副词、"清"作为形容词前缀）
  - 判断方法：如果该子串出现在对话中作为称呼、或紧跟在姓氏后面 → 标记为泄漏
  - 如果该子串是常见汉语词汇且用法符合语境 → 跳过
如果原书无 character-map.md → 跳过此维度，标记 N/A
```

**维度 9：伏笔回收质量评估**（仅 Suggestion，不阻断）
```
对 foreshadowing.md 中标记"已回收"的伏笔，读取回收章节相关段落
按3项打分（每项1分）：
  ① 是否明确点名回收该伏笔（1分）
  ② 是否推动后续情节（1分）
  ③ 是否带来情绪/关系变化（1分）
3分=A级（优秀），2分=B级（合格），0-1分=C级（不足）
C级 → Suggestion："伏笔回收力度不足，建议扩写"
```
