# Phase 2：骨架提取 + 压缩规划（核心阶段）

**进入时**：如果 novels/{name}/ 目录已存在（中断恢复场景），跳过目录创建；否则创建 novels/{name}/ 并写入 meta.json（`status: "planning"`, `mode: "short"`）使恢复分支可达。

**前置条件**：source 的 analysis.md 存在且包含压缩标注，source meta.json 包含 `word_count` 字段

**前置检查**：
```
IF source meta.json.word_count <= 45000:
  → 终止，提示"原作字数 ≤ 45000，无需压缩，建议使用 /novel 进行等长仿写"
IF source meta.json.word_count <= 50000:
  → 警告"原作字数接近目标下限，压缩空间有限"，但继续执行
```

## Step 1：情节骨架提取

读取 analysis.md 的全部章节标注，提取情节骨架。

**输出文件：`novels/{name}/config/plot-skeleton.md`**

```markdown
# 情节骨架

## 原作概况
- 原作：{source_title}
- 总章数：{N}章
- 总字数：{N}字
- 目标字数：45000-50000字（硬上限 50000）

## 主线弧线
（按起承转合描述核心主线，每个节点标注原作章节号）
1. 【起】第X-Y章：{开局描述}
2. 【承】第X-Y章：{发展描述}
```

## Step 2：生成压缩映射表

基于骨架，计算压缩方案。

**计算逻辑**：
```
source_words = meta.json.word_count
hard_limit = 50000
gemini_inflation = 1.17       # v5 实测：Gemini Pro 平均输出目标字数的 117%
budget_words = round(hard_limit / gemini_inflation)   # ≈ 42735，考虑 Gemini 膨胀后刚好达到 50K

IF budget_words / 1500 > 40:
    chapters = round(budget_words / 1500)
ELSE IF budget_words / 1500 < 25:
    chapters = round(budget_words / 1200)
ELSE:
    chapters = round(budget_words / 1400)

avg_words_per_chapter = round(budget_words / chapters)
```

**⚠️ 膨胀率校准（v5 实测数据）**：
Gemini Pro + maxOutputTokens=target*2.5 时，实际输出平均为目标的 117%。
因此 compression-map 的目标字数之和应为 ~42700 而非 45000-50000。
写入 meta.json 时：target_words=50000, budget_words=42700。

**字数控制原则**：
- `target_words`（50000）是硬上限，成稿总字数绝不能超过
- `budget_words`（45000）是实际分配预算
- 10% 余量用于吸收并行写作时各章±10%容差的累积超标

**输出文件：`novels/{name}/config/compression-map.md`**

```markdown
# 压缩映射表

## 压缩参数
- 原作字数：{source_words}
- 硬上限：50,000 字
- 分配预算：{budget_words} 字
- 压缩比：{ratio}:1
- 新作章数：{chapters}章
- 平均单章字数：约{avg}字

## 映射表

**事件 ID 编号规则**：`E{新作章号}_{序号}`，如 E01_01、E01_02、E02_01。
每章的核心事件必须编号，相邻章的事件 ID 不得重复占用。

| 新作章 | 对应原作章 | 压缩策略 | 目标字数 | 核心事件ID列表 | 起始事件ID | 终止事件ID |
|--------|-----------|---------|---------|--------------|-----------|-----------|
| 示例 | 原1-3 | 合并 | 1500 | E01_01,E01_02,E01_03 | E01_01 冰柜重生 | E01_03 获九霄塔 |

**合计分配预算：{budget_total} 字 | 硬上限：50,000 字**

## 事件边界规则（防并行写作重叠）
- 每章必须标注起始/终止事件 ID，Agent 只负责自己的事件集合
- 相邻章的终止事件 ID 必须在下一章起始事件 ID 之前（时序不倒挂）
- 相邻章的核心事件 ID 列表不得有交集（无所有权重叠）
- 连续场景跨章时，必须明确切分点："本章止于[事件]，下章始于[事件]"

## 压缩策略说明
- **等长改写**：原作章节内容密度高，基本1:1改写
- **合并**：多章合并为一章，保留核心事件，删除过渡和灌水
- **删支线，保主线**：只保留主线相关内容
- **浓缩**：保留所有要素但压缩描写篇幅
```

## Step 3：启动校验 Agent

生成 plot-skeleton.md 和 compression-map.md 后，**启动独立校验 Agent**（subagent_type: code-reviewer）。

**校验 Agent 执行 8 项校验**：

1. **字数校验**：所有章节目标字数之和 ≥ 45000 且 ≤ 50000
2. **覆盖校验**：原作所有章节都被映射到
3. **伏笔链校验**：所有伏笔的埋设章和回收章都在映射范围内
4. **节奏校验**：前 20% 章节占 15-20% 字数，最后 10% 占 8-15%
5. **不可压缩章节校验**：标记为"不可压缩"的章节全部保留
6. **因果链校验**：被依赖章节的核心事件在映射表中被包含
7. **单章映射密度校验**：不超过 8 章/新章
8. **事件连续性校验**（防并行写作重叠）：
   FOR 每对相邻新章 (N, N+1)：
   - N 的终止事件 ID 与 N+1 的起始事件 ID 之间无事件断档
   - N 的核心事件 ID ∩ N+1 的核心事件 ID == ∅（无所有权重叠）
   - 如有交集 → 校验 Agent 重新划分事件归属

**校验结果处理**：
```
IF 发现问题 → 校验 Agent 直接修正，最多 3 轮
IF 全部通过 → 在 compression-map.md 末尾追加校验通过记录
```

**转场**：校验通过后，更新 meta.json status → "configuring"，自动进入 Phase 3
