# Phase 1：分析（骨架提取 + 章节分级）

## 前置条件

- 用户指定或自动选择一本 `status: "complete"` 的小说（记为 `source_novel`）

## Step 1：精确统计

按 CLAUDE.md "字数统计规范"逐章统计字数：

```
遍历 novels/{source_novel}/chapters/*.md
每章：读取全文 → 去除 Markdown 标记 → 去除所有空白字符 → 计算字符数
汇总：original_total = 各章字数之和
```

确定目标字数（`target_total`）：
- `/compress auto` 模式：自动采用默认值 50000，不等待用户输入
- 非 auto 模式（如 `/compress analyze`）：提示用户确认，默认 50000

计算需削减总量 = `original_total - target_total`。

如果 `original_total ≤ target_total`：提示"原稿字数已在目标范围内，无需压缩"，终止。
如果削减比例 < 30%：建议使用 `/condense` 替代（精炼更适合小幅压缩）。

## Step 2：逐章分析

对每章标注：
- **章节功能**：主线推进 / 支线展开 / 人物塑造 / 世界观铺垫 / 过渡灌水
- **可压缩度**：A(高) / B(中) / C(低) / D(不可压缩)
- **依赖关系**：被哪些后续章节依赖（因果链）
- **伏笔关联**：埋设/回收了哪些伏笔
- **是否可删除**：标记纯支线/灌水章节为"可删除"（主线无依赖）

## Step 3：骨架提取

从全书中提取：
- **主线弧线**：核心事件链（必须保留）
- **支线清单**：每条支线的起止章节、与主线的关联度
- **支线取舍建议**：标记"必留"/"可删"/"可浓缩"
- **角色出场清单**：每个角色的首次出场章、关键章节、是否可删除

## Step 4：事件清单（每章 3-8 要点）

逐章提取不可删除的核心事件：

- 关键事件（推动情节的动作/决策）
- 角色关系变化（结盟/反目/暴露身份）
- 伏笔埋设/回收点
- 因果节点（A 导致 B 的关键连接）
- 情绪转折点
- 章尾钩子

## Step 5：伏笔校准

```
0. 检查原书 foreshadowing.md 状态
   IF 文件不存在：
     → 检查原书 memory-outline.md 的"活跃伏笔列表"
     → IF memory-outline 有伏笔条目：从 memory-outline + 章节扫描回填，生成伏笔表后继续正常校准
     → IF memory-outline 也无伏笔：标注"原书无伏笔记录"，创建空占位文件，后续标记 N/A，直接进入 Step 6
   IF 文件存在但无数据行（只有表头）：
     → 检查原书 memory-outline.md 的"活跃伏笔列表"
     → IF memory-outline 有伏笔条目：视为空表异常，从 memory-outline + 章节扫描回填伏笔表到 compress-plan.md 中，然后继续正常校准
     → IF memory-outline 也无伏笔：标注"原书伏笔表和记忆均为空"，创建空占位文件，后续标记 N/A，直接进入 Step 6
1. 读取伏笔数据源（原书 foreshadowing.md 或上方回填生成的版本）
2. 逐条在实际章节中搜索确认：
   - 埋设事件是否在标注的章节中存在
   - 回收事件是否在标注的章节中存在
3. 标记脏数据（表中有记录但章节中找不到对应内容）
4. 生成校准后的伏笔状态表，写入 compress-plan.md
```

## 输出

- 创建 `novels/{name}-compressed/` 目录结构（仅 config/ 目录）
- 写入 compress-plan.md 的"分析"部分（章节分析表 + 骨架 + 支线清单 + 事件清单 + 伏笔校准表）
- 创建初始 meta.json
- 更新 meta.json：`status → "analyzed"`

**转场**：自动进入 Phase 2。

# Phase 2：压缩规划

## 前置条件

meta.json `status` 为 `"analyzed"`

## Step 1：生成压缩映射表

核心决策逻辑：
```
1. 识别可删除章节（纯支线/灌水，对主线无因果依赖）→ 标记为"删除"
2. 识别可合并章节组（相邻 + 功能相近 + 合并后因果链不断）
3. 识别需保留章节（高潮转折、不可压缩、关键伏笔回收）→ 标记为"保留"或"精炼"
4. 将剩余章节按相邻关系分组合并
5. 硬约束：
   - 单章映射密度 ≤ 5 章（不能把太多原作章节塞进一章）
   - 任何新章目标字数 ≥ 800（保证逻辑完整性）
   - 所有伏笔的埋设章和回收章都必须被映射覆盖
   - 所有主线事件都必须在某个新章中保留
```

映射表格式：

```markdown
## 压缩映射表

**事件 ID 编号规则**：`E{新章号}_{序号}`，如 E01_01、E01_02。
每章核心事件必须编号，相邻章的事件 ID 不得重复占用。

| 新章号 | 对应原作章 | 压缩策略 | 目标字数 | 核心事件ID列表 | 起始事件ID | 终止事件ID |
|--------|-----------|---------|---------|--------------|-----------|-----------|
| 1 | 原1-3章 | 合并+精炼 | 1500 | E01_01,E01_02,E01_03 | E01_01 事件A | E01_03 事件C |
| 2 | 原4-5章 | 合并 | 1400 | E02_01,E02_02 | E02_01 事件D | E02_02 事件E |
| - | 原6章 | 删除 | - | — | — | — |

**事件边界规则**（防并行写作重叠）：
- 每章必须标注起始/终止事件 ID，Agent 只负责自己的事件集合
- 相邻章的核心事件 ID 列表不得有交集
- 连续场景跨章时，必须明确切分点
```

## Step 2：字数预算分配

```
target_words = 用户指定（默认 50000）
budget_words = target_words × 0.95（安全余量，确保不超标）

计算新章总数 = 映射表中非"删除"行数
平均单章字数 = budget_words / 新章总数

按压缩策略分配：
- "保留/精炼"类：目标 = 原章字数 × 0.85（轻微精炼）
- "合并"类：目标 = 合并源章字数总和 × 压缩比（由章数和内容密度决定）
- "浓缩"类：目标 = 原章字数 × 0.50-0.70

硬约束：
- 所有新章目标之和 = budget_words（±2%）
- 硬上限：所有新章目标之和 ≤ target_words（绝不超过目标字数）
```

## Step 3：伏笔重映射

如果 Phase 1 标注"原书无伏笔表，伏笔校准已跳过"，则跳过本步骤（占位文件已在 Phase 1 创建，无需重映射）。

```
遍历校准后的伏笔表：
  对每条伏笔：
    - 原埋设章 X → 在映射表中找到包含 X 的新章号 → 更新为新章号
    - 原回收章 Y → 在映射表中找到包含 Y 的新章号 → 更新为新章号
    - 如果原章被标记为"删除"：
      → 检查是否有伏笔依赖该章
      → 如有，按方向性约束迁移：
        - 埋设内容 → 向前迁移（迁移到更早的相邻保留章），确保先埋后收
        - 回收内容 → 向后迁移（迁移到更晚的相邻保留章），确保回收在埋设之后
      → 迁移后验证：每条伏笔的埋设新章号 < 回收新章号（时序不倒挂）
      → 更新映射表，确保无伏笔丢失
写入 novels/{name}-compressed/config/foreshadowing.md（重映射后的版本）
```

## Step 4：记忆大纲重建

```
读取原书 memory-outline.md + compress-plan.md 的事件清单和映射表
不做简单章号替换（合并/删除会导致章号语义变化），而是从头重建：
  1. 按新章号顺序，从 compress-plan.md 的事件清单中提取每章核心事件
  2. 对于合并章（多原章→1新章）：合并多章事件为一条摘要
  3. 对于删除章：跳过（其关键内容已迁移到保留章）
  4. 保留原书 memory-outline 中的世界规则、角色关系等全局信息（不含章号引用）
  5. 确保重建后的大纲 ≤ 2000 字
写入 novels/{name}-compressed/context/memory-outline.md
```

## Step 5：独立 Agent 校验

启动独立校验 Agent（subagent_type: code-reviewer），执行 7 项校验：

1. **字数校验**：所有新章目标之和 ≤ target_words 且 ≥ target_words × 0.95
2. **覆盖校验**：每个原作主线章节都被映射覆盖（标记为"删除"的支线章除外）
3. **伏笔链校验**：所有伏笔的埋设章和回收章都在映射范围内
4. **节奏校验**：前20%章节占15-25%字数，末10%章节占8-15%字数
5. **因果链校验**：每个"被依赖"章节的核心事件在新章中保留
6. **单章密度校验**：无新章映射超过5个原作章节
7. **事件连续性校验**（防并行写作重叠）：
   FOR 每对相邻新章 (N, N+1)：
   - N 的终止事件 ID 与 N+1 的起始事件 ID 之间无事件断档
   - N 的核心事件 ID ∩ N+1 的核心事件 ID == ∅（无所有权重叠）
   - 如有交集 → 校验 Agent 重新划分事件归属

**校验结果处理**：
```
IF 发现问题：
  → 校验 Agent 直接修正映射表
  → 修正后再执行一轮校验
  → 最多 3 轮修正，3 轮后仍有问题则记录到日志继续
IF 全部通过：
  → 在 compress-plan.md 末尾追加校验通过记录
```

## 输出

- compress-plan.md 完成（分析部分 + 映射表 + 预算 + 伏笔重映射 + 校验记录）
- 创建完整目录结构：
  ```
  novels/{name}-compressed/
  ├── meta.json
  ├── config/
  │   ├── compress-plan.md
  │   └── foreshadowing.md        ← 重映射后的版本
  ├── context/
  │   ├── memory-outline.md       ← 从事件清单重建
  │   ├── recent-context.md       ← 空文件
  │   └── archives/               ← 空目录
  └── chapters/                   ← 空目录
  ```
- 更新 meta.json：`status → "planned"`，`compressed_chapters = 新章总数`

**转场**：自动进入 Phase 3。
