# /short Skill 改进方案

## 背景

《野狗不识字，但野狗护你》完成后，经 Code Review Agent + Codex CLI 两路独立审核，发现 9 个 Critical + 12 个 Warning 级别的逻辑一致性问题。根因分析显示问题集中在并行写作阶段的质量控制缺失。

本文档汇总全部改进点，覆盖两路审核发现 + Codex CLI 架构建议，目标是从根本上解决"并行写作导致的逻辑不一致"问题。

---

## 一、问题根因（5条）

| # | 根因 | 导致的典型问题 |
|---|------|--------------|
| R1 | 并行 Agent 无统一事实源，各自编造细节 | 楼层47/37、断指左/右手、身世两版本、年龄23/24、股权7%/12% |
| R2 | 终验只抽查不通读，跳过连续性检查 | 季天成被抓两次（ch20→21→22时间线）、场景回跳（ch10→11） |
| R3 | 审校触发不足（设计每5章，实际只跑1次） | ch9-32共24章零审校 |
| R4 | 只做泄漏扫描不做内部一致性扫描 | 同一细节跨章矛盾全部漏检 |
| R5 | 批次间重叠扫描只查事件ID，不查叙事状态 | "已被拘留"的人出现在晚宴、"逃离老宅"后又回到保险柜前 |

## 二、改进方案（7条）

### 方案 A：新增"事实锁定表" (details-lock.md)

**对应根因**：R1

**改动位置**：Phase 3（配置阶段）新增产出物

**内容**：
```markdown
# 事实锁定表

## 人物物理细节
| 角色 | 细节项 | 锁定值 | 首次出现章 |
|------|--------|--------|-----------|
| 季野 | 断指位置 | 右手小指缺半截 | 新02 |
| 季野 | 断指原因 | 季天成用钳子掰断，5岁时 | 新11揭露 |
| 沈念 | 年龄 | 23岁（重生时）→故事跨度约9个月→后期仍23岁 | 新01 |

## 场景/地点固定参数
| 地点 | 细节项 | 锁定值 |
|------|--------|--------|
| 天成集团大楼 | 楼层 | 37层 |
| 念园 | 首次命名章 | 新27（之前只能用"安全屋""公寓"） |

## 数字/比例
| 项目 | 锁定值 | 涉及章节 |
|------|--------|---------|
| 沈念持股 | 12%（被转让前）→ 4%（被转让后）→ 信托释放21% | 新28/29 |

## 身世统一版本
| 事件 | 锁定版本 |
|------|---------|
| 季野童年 | 0-6岁关在地下室 → 6岁被装塑料袋扔垃圾站 → 被收破烂老头捡回养大 |
| 断指真相 | 季天成所为（钳子），5岁时 |
| 傅雪结局 | 产后精神障碍+被虐待 → 精神崩溃后伤害幼子 → 最终死于季天成控制下（具体方式由ch22录音揭露）|
```

**规则**：
- Phase 3 创建此表，与 character-map / setting-map 同级
- 所有写作 Agent 的 prompt 必须包含此表
- Agent 不得编造表中未列的具体数字/物理细节；如需新增，必须先追加到此表
- 表有版本号，每次修改记录变更原因

**Codex CLI 补强建议落地**：
- 表头增加 `version` 字段
- 每次修改追加变更日志（谁改了什么、为什么）
- 修改后自动定位受影响的已写章节（通过 Grep 关键词）

### 方案 B：新增"事件账本" (event-ledger.md)

**对应根因**：R5（Codex CLI 指出的遗漏）

**改动位置**：Phase 4 每章写完后自动追加

**内容**：
```markdown
# 事件账本

| 章号 | 事件 | 人物状态变更 | 地点 | 时间标记 |
|------|------|------------|------|---------|
| 01 | 沈念重生，坐上奔驰赴约 | 沈念：自由→赴约途中 | 沈家大宅→奔驰车上 | 订婚宴当晚 |
| 02 | 季野拦车劫持沈念 | 沈念：赴约→被劫持；季野：自由 | 路上→城中村出租屋 | 同晚 |
| ... | ... | ... | ... | ... |
| 20 | 商业围剿+检察院立案 | 季天成：自由→资产冻结（人未被抓） | 季家庄园 | 执行日周五 |
| 21 | 姚光晚宴揭发+法律缠斗 | 季天成：自由但被围剿 | 滨海商会→季家庄园 | 冻结后48小时内 |
| 22 | 废弃工厂伏击，季天成被捕 | 季天成：自由→被逮捕 | 废弃工厂 | 晚宴后数日 |
```

**规则**：
- 每章写完后，Agent 必须输出一条 `chapter_delta`（结构化的状态变更记录），追加到此表
- 并行模式下，每批次完成后统一回填
- **状态连续性扫描（方案E）读取此表作为校验依据**
- 固定状态字段（Codex 建议的6-8个）：
  1. 地点（在哪里）
  2. 人身自由（自由/被软禁/被拘留/在逃）
  3. 伤情（健康/受伤/PTSD发作中）
  4. 认知状态（知道某事/不知道某事）
  5. 关系状态（敌对/中立/联盟/恋人）
  6. 资产状态（正常/冻结/转移）

### 方案 C：并行模式改为"波次并行"

**对应根因**：R1、R5

**改动位置**：writing-loop.md 的"执行模式选择"部分

**当前设计**：固定批次，奇数章→偶数章→...

**改进设计**：

```
Phase 4 执行模式：

1. 串行主干期（前4章必须串行）
   - ch1→ch2→ch3→ch4 严格串行
   - 目的：建立核心事实基线、填充 details-lock.md 和 event-ledger.md
   - 完成后：details-lock.md 中的核心细节已固化

2. 波次并行期（第5章起）
   - 按"依赖图"分波次，不按固定奇偶分批
   - 依赖图规则：
     a. 强耦合章节（同一时间线上的连续事件）必须在同一波次内串行
        示例：ch20(围剿)→ch21(晚宴)→ch22(逮捕) 是强耦合，不可拆分到不同波次
     b. 弱耦合章节（时间跳跃、场景切换）可并行
        示例：ch5(城中村日常) 和 ch7(老街偶遇) 可并行
   - 识别强耦合的方法：
     压缩映射表中，如果两章的终止事件和起始事件涉及同一角色的连续状态变更
     （如"被抓→出庭→定罪"），标记为强耦合
   - 每波次仍不超过4-5个 Agent

3. 每波次完成后必须执行（硬门禁，不可跳过）：
   a. 状态连续性扫描（方案E）
   b. 事实一致性扫描（方案D）
   c. 审校触发检查
   d. 以上任一发现 Critical → 阻断，修复后才能进入下一波次
```

**Codex CLI 建议落地**：
- "先串行产出1-4章主干，再并行扩写" → 已纳入
- "按依赖图并行，不按固定批次" → 已纳入
- "并行保留，但每章必须读入最新状态快照" → 通过 event-ledger.md 实现

### 方案 D：事实一致性自动扫描

**对应根因**：R4

**改动位置**：
- writing-loop.md 每批次后处理步骤（新增）
- Phase 7 终验（增强）

**扫描逻辑**：
```
1. 从 details-lock.md 提取所有锁定值的关键词
   例如："右手小指""37层""12%""23岁""安全屋"（ch27前）/"念园"（ch27后）

2. 用 Grep 跨全部已写章节扫描每个关键词的所有出现

3. 规则比对：
   - "右手小指" 出现的地方不应该同时存在 "左手小指"
   - "37层" 出现的地方不应该同时存在 "47层"
   - "念园" 在 ch27 之前的章节中不应该出现
   - 年龄/股权等数字：同一角色在同一时间段内的数值必须一致

4. 输出冲突报告：
   | 关键词 | 锁定值 | 冲突出现 | 章节:行号 | 严重度 |
   |--------|--------|---------|----------|--------|

5. Critical 冲突 → 阻断进入下一波次
```

**Codex CLI 补强建议落地**：
- Grep 只做第一层召回（找到所有出现位置）
- 第二层做"归一化比对"（同义词、代词识别）
- 例如："季野的断指""他缺了半截的小指""那只残缺的手" 都指向同一个事实

### 方案 E：状态连续性扫描

**对应根因**：R5

**改动位置**：writing-loop.md 每批次后处理步骤（新增，在方案D之后执行）

**扫描逻辑**：
```
读取 event-ledger.md，逐对检查相邻章节：

FOR i = 1 to current_chapter - 1:
  ch_i_end_state = event-ledger 中第 i 章的结束状态
  ch_i1_start = 第 i+1 章开头描述的状态

  检查以下6个维度：
  1. 地点连续性：ch_i 结尾在A地 → ch_i+1 开头应在A地或有明确的转场
  2. 人身自由：ch_i 结尾被拘留 → ch_i+1 不应该自由出席活动（除非有释放描写）
  3. 伤情连续性：ch_i 受重伤 → ch_i+1 不应该生龙活虎（除非有时间跳跃或治疗）
  4. 认知连续性：ch_i 角色还不知道某事 → ch_i+1 不应该突然知道（除非有告知场景）
  5. 关系连续性：ch_i 两人还是敌对 → ch_i+1 不应该突然是盟友
  6. 道具/资产连续性：ch_i 拿走了X → ch_i+1 不应该说X还在原处

  发现断裂 → 标记为 Critical，输出：
  "ch{i}→ch{i+1} 状态断裂：{维度}，ch{i}结尾={A}，ch{i+1}开头={B}"
```

**Codex CLI 建议落地**：
- 先固定6个硬状态字段 → 已纳入
- 状态定义必须清晰，避免规则失效 → 通过 event-ledger 的结构化字段保证

### 方案 F：审校强制触发 + Critical 硬门禁

**对应根因**：R3

**改动位置**：writing-loop.md 每批次后处理步骤（强化现有设计）

**当前设计**：每5章触发审校（但实际执行被跳过）

**改进设计**：
```
审校触发规则（不可跳过，硬编码）：

1. 每批次完成后，无条件触发审校
   - 范围 = 本批次全部章节 + 前后各1章（检查衔接）
   - 不再依赖"每5章"计数器（因为并行模式下章号不连续）

2. 审校检查项增加（在原有6项基础上新增）：
   - [新增] details-lock.md 一致性（方案D的结果）
   - [新增] event-ledger.md 状态连续性（方案E的结果）
   - [新增] 场景转换自然度（上章结尾→本章开头是否有过渡）

3. Critical 硬门禁：
   - 审校发现 Critical 级别问题 → 阻断，不允许进入下一批次
   - 必须修复后复检通过才能继续
   - Warning 级别 → 记录，可延后到全书完成后修复
   - Suggestion 级别 → 记录，不阻断

4. 审校结果写入 review-notes.md（追加模式），格式：
   ```
   ## 审校：批次 {N}，范围 ch{X}-ch{Y}
   时间：{timestamp}
   Critical: {count}
   Warning: {count}
   [详细问题列表]
   ```
```

### 方案 G：终验改为强制顺序通读

**对应根因**：R2

**改动位置**：Phase 7（verification.md）

**当前设计**：可抽查

**改进设计**：
```
终验执行规则（不可简化）：

1. 必须逐章顺序阅读，不允许抽查模式
   - 如果 context 不够一次读完，分批：ch1-10, ch11-20, ch21-32
   - 每批内必须按顺序读，不可跳章

2. 逐对衔接检查（ch1→2, ch2→3, ... ch31→32）：
   - 场景是否连贯（上章结尾地点 = 下章开头地点）
   - 人物状态是否连续
   - 时间线是否合理
   - 有无"跳跃感"

3. 终验必须消费前序阶段的报告：
   - 读取 review-notes.md（全部审校记录）
   - 读取 event-ledger.md（事件账本）
   - 读取 details-lock.md（事实锁定表）
   - 对照检查，而不是纯人工重读

4. 终验新增维度：
   - [新增] 跨章数字一致性（从 details-lock.md 校验）
   - [新增] 跨章状态连续性（从 event-ledger.md 校验）
   - [新增] 道具追踪（提到的物品是否有头有尾）
   - [新增] 反派智商审计（反派的行为是否符合其设定的能力水平）
```

## 三、章节输出契约（Codex CLI 指出的遗漏）

每个写作 Agent 完成章节后，除了输出章节正文，还必须输出结构化的 `chapter_delta`：

```json
{
  "chapter": 20,
  "events": ["商业围剿启动", "十七家壳公司发函", "检察院立案", "资产冻结"],
  "state_changes": {
    "季天成": {"人身自由": "自由→资产冻结但人未被抓", "资产": "正常→全线冻结"},
    "沈念": {"地点": "安全屋", "认知": "知道壳公司全貌"}
  },
  "new_facts": {
    "季天成反应": "没慌，发朋友圈'小把戏'"
  },
  "end_location": {"沈念": "安全屋", "季野": "安全屋", "季天成": "季家庄园"},
  "time_marker": "执行日周五晚"
}
```

此 delta 用于：
- 自动更新 event-ledger.md
- 供下一章 Agent 作为输入（替代"只读前章结尾"）
- 状态连续性扫描的数据源

## 四、优先级冲突规则（Codex CLI 指出的遗漏）

当以下信息源冲突时，按此优先级裁决：

```
details-lock.md（事实锁定表）> event-ledger.md（事件账本）> 前章正文 > 原作内容 > Agent 自行编造

即：
1. 如果锁定表说"右手小指"，无论原作怎么写、前章怎么写，都用"右手小指"
2. 如果事件账本说"季天成此时未被抓"，Agent 不得写他被拘留
3. 如果前章正文和锁定表冲突，以锁定表为准，并标记前章需修复
4. Agent 绝不可自行编造锁定表中未列的具体数字/物理细节
```
