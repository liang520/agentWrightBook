# /short Skill 改进方案 v3

> v2 → v3 变更说明：纳入第三轮审核——Codex CLI（2个未闭环+1个示例矛盾）+ Skill Creator（3 Critical + 3 Important + 3 Suggestion）的全部意见共11处修复。标记 [v3] 为本轮新增/修改内容。

> 角色定义 [v3]：本方案中"协调进程"= 执行 /short auto 的主 Agent。"写作 Agent"和"提取 Agent"都是由主 Agent 通过 subagent 调用的子任务。三者不是独立服务。

## 背景

《野狗不识字，但野狗护你》完成后，经 Code Review Agent + Codex CLI 两路独立审核，发现 9 个 Critical + 12 个 Warning 级别的逻辑一致性问题。根因分析显示问题集中在并行写作阶段的质量控制缺失。

本文档汇总全部改进点，覆盖两路审核发现 + Codex CLI 架构建议 + 两路方案审核反馈，目标是从根本上解决"并行写作导致的逻辑不一致"问题。

- Phase 3 创建此表，与 character-map / setting-map 同级
- 写作 Agent 的 prompt 分层加载此表（不是全量加载）[v3]：
  - 加载：与当前章涉及的角色相关的物理细节条目
  - 加载：角色身份阶段表（全量，通常很短）
  - 加载：当前章涉及的道具归属条目
  - 不加载：与当前章无关的角色/道具条目
  - 全量加载仅在：波次后处理扫描、终验
- Agent 不得编造表中未列的具体数字/物理细节
- 需要新数字时：写作 Agent 在正文中使用**模糊措辞**（如"那栋楼"而不是"47层"）。提取 Agent 会在 chapter_delta 的 `new_facts_to_lock` 中标记为"待锁定"。写作 Agent 本身不输出任何 JSON。 [v3]
- 每波次后处理时统一裁决待锁定事实，写入此表，回填章节中的模糊措辞 [v2-new]

**details-lock 变更治理流程** [v2-new]：
```
details-lock 变更 → Grep（使用搜索关键词列表）定位受影响章节 → 分级处理：
  - 简单替换型（如"左手"→"右手"）：
    → 自动替换 + 记录到变更日志 + 不阻断
  - 语义变更型（如改变断指原因从"烟头"到"钳子"）：
    → 标记为 Blocking Critical，阻断当前波次
    → 人工/Agent 改写受影响段落，确保语义通顺
    → 改写后复检通过才继续
  - 未来章节影响型（如改了尚未写到的伏笔回收方式）：
    → 更新 foreshadowing.md + compression-map.md
    → 不阻断当前波次，但标记"后续写作需注意"
```

### 方案 B：新增"事件账本" (event-ledger.md)

**对应根因**：R5

**改动位置**：Phase 4 每章写完后追加

**内容**：
```markdown
# 事件账本

| 章号 | 事件 | 人物状态变更 | 地点 | 时间标记 |
|------|------|------------|------|---------|
| 01 | 沈念重生，坐上奔驰赴约 | 沈念：自由→赴约途中 | 沈家大宅→奔驰车上 | 订婚宴当晚 |
| 02 | 季野拦车劫持沈念 | 沈念：赴约→被劫持；季野：自由 | 路上→城中村出租屋 | 同晚 |
| ... | ... | ... | ... | ... |
```

**状态字段（6个硬字段）**：
1. 地点（在哪里）
2. 人身自由（自由/被软禁/被拘留/在逃）
3. 伤情（健康/受伤/PTSD发作中）
4. 认知状态（知道某事/不知道某事）
5. 关系状态（敌对/中立/联盟/恋人）
6. 资产状态（正常/冻结/转移）

**chapter_delta 生成与写入机制** [v2→v2.1 统一，v2.1-eng 去重]：
```
职责划分和提取 Agent 的完整规格详见第三节"章节输出契约"（单一信息源）。

本节只描述并发安全和回滚机制：

写入流程（主 Agent 负责）[v3 术语修正]：
  1. 波次内所有写作 Agent 完成，输出章节正文到 chapters/{NNN}.md
  2. 主 Agent 为本波次每章调用提取 Agent（规格见第三节）
  3. 主 Agent 按章号顺序读取全部 .delta.json
  4. 复制 event-ledger.md 为 event-ledger.prev.md（波次前版本）
  5. 在 meta.json 中记录 ledger_version += 1
  6. 按章号顺序合并写入 event-ledger.md
  7. 执行状态连续性扫描（方案 E）
  8. 如扫描发现 Blocking Critical → 触发恢复（见下方）

恢复机制 [v3 改用 .prev 术语]：
  - 合并前保存 event-ledger.prev.md
  - 扫描失败 → 用 event-ledger.prev.md 覆盖 event-ledger.md
  - meta.json 中 ledger_version 回退
  - 删除本波次的 .delta.json
  - 修复问题章节正文后，重新提取 delta → 重新合并 → 重新扫描
  - 最多恢复 2 轮，之后降级为串行模式
```

**event-ledger 加载规则**（预留长篇扩展）[v2-new]：
```
- 短篇（≤50章）：写作 Agent 加载全量 event-ledger
- 长篇（>50章）：写作 Agent 只加载最近20条 + 涉及当前章核心角色的历史条目
- 终验和状态扫描始终读取全量
```

### 方案 C：并行模式改为"波次并行"

**对应根因**：R1、R5

**改动位置**：writing-loop.md 的"执行模式选择"部分

**改进设计**：

```
Phase 4 执行模式：

1. 串行主干期（前4章必须串行）
   - ch1→ch2→ch3→ch4 严格串行
   - 目的：建立核心事实基线，填充 details-lock.md 和 event-ledger.md
   - 完成后：details-lock 中的核心细节已固化

2. 波次并行期（第5章起）
   - 按"依赖图"分波次，不按固定奇偶分批
   - 强耦合识别规则（含依赖图构建算法）[v2-changed]：

     输入：compression-map.md（每章的核心事件ID+起止事件+角色列表）

     波次分配规则（启发式，Agent 可直接执行）[v3 重写，取消拓扑排序]：
     ```
     主 Agent 按以下步骤分配波次（读表→判断→写表，无需图算法）：

     步骤1：标记强耦合对
       扫描 compression-map，逐对检查相邻章(ch_i, ch_i+1)：
         IF ch_i 终止事件涉及角色A的状态变更（被抓/受伤/暴露/死亡）
            AND ch_i+1 起始事件涉及同一角色A
         → 标记 (ch_i, ch_i+1) 为强耦合

       扫描 foreshadowing.md（含承诺追踪表）：
         IF 埋设章与回收章间距 ≤ 5 → 标记为强耦合

       扫描 details-lock.md 道具归属表：
         IF 同一道具的两个转移章节间距 ≤ 5 → 标记为强耦合
         IF 间距 > 5 → 不标记，靠方案E全局约束扫描兜底

     步骤2：归组
       将所有强耦合对合并为"串行组"（传递闭包：A-B强耦合、B-C强耦合 → A-B-C同组）
       组内按章号排序，标记为"串行"

     步骤3：填入波次
       依次处理每个串行组和剩余独立章节：
         - 串行组整体放入一个波次，组内严格串行
         - 独立章节（无强耦合关系）填入最早的可用波次
         - 每波次上限 4-5 章（含串行组占用的位置）
         - 如果某串行组超过 5 章，该组独占一个波次

     步骤4：输出 context/wave-plan.md
       | 波次 | 章号列表 | 模式 | 强耦合原因 |
       |------|---------|------|-----------|
       | 1    | 5,7     | 并行 | 无 |
       | 2    | 6,8     | 并行 | 无 |
       | 3    | 9,10    | 串行 | F06伏笔链(9→10) |
       | 4    | 11,13   | 并行 | 无 |
       | ...  | ...     | ...  | ... |
       | 8    | 20,21,22| 串行 | 季天成状态连续变更 |
     ```

     示例1（相邻强耦合）：ch20(围剿)→ch21(晚宴)→ch22(逮捕)
     → 同一角色连续状态变更 → 强耦合 → 同一波次内串行

     示例2（道具链弱耦合，距离>5）：ch10(取出傅雪录音)→ch22(播放傅雪录音)
     → 距离12章 > 阈值5 → 不建强耦合，靠方案E全局约束扫描兜底 [v3 修正]

     示例3（伏笔链强耦合）：F06 保险柜 ch09(埋设)→ch10(回收)
     → 距离1章 ≤ 5 → 强耦合 → 同一波次

   - 每波次仍不超过4-5个 Agent
   - [v3] 写作中新发现的依赖：提取 Agent 从正文中识别到引用了尚未写的后续章节内容时，
     在 chapter_delta.new_dependencies 中标记（如 ["ch20→ch22"]），
     主 Agent 在下一波次分配时纳入考虑。写作 Agent 本身不标记任何 JSON 字段。
   - [v2.1-new] 前置防错仍有漏检窗口的兜底：
     即使依赖图未能识别所有强耦合，方案 E 的全局状态约束扫描
     会在波次完成后捕获非相邻章的状态矛盾，作为第二道防线

3. 保留现有成熟机制 [v2-new]：
   波次并行是对现有并行模式的升级替换，但以下机制必须保留：
   - 相邻章重叠扫描（事件ID + 文本场景比对）
   - 降级串行（重叠检测失败后自动降级）
   - 前章注入（读取前章全文或尾部）
   - 防越界硬指令（4条规则原样保留）
   - 前章不存在时的过渡指令

4. 每波次完成后必须执行（硬门禁，不可跳过，严格串行）[v2.1-changed]：
   a. 提取 chapter_delta（方案B的提取Agent，规格详见第三节）
   b. 合并写入 event-ledger.md              ← 前置：a 全部完成
   c. 锁定待定事实（汇总 new_facts_to_lock） ← 前置：b 完成（ledger 已更新）
   d. 事实一致性扫描（方案D）                ← 前置：c 完成（details-lock 已更新）
   e. 状态连续性扫描（方案E）                ← 前置：b 完成（ledger 可读）
   f. 审校触发（方案F）                      ← 前置：d + e 完成（消费扫描结果）
   g. 门禁判定                               ← 前置：f 完成
      d/e/f 任一发现 Blocking Critical → 修复后才能进入下一波次
   注意：d 和 e 可并行执行（两者都只读 ledger 和 lock，无写冲突），其余必须串行。
```

### 方案 D：事实一致性自动扫描

**对应根因**：R4

**改动位置**：波次后处理步骤 + Phase 7 终验

**扫描逻辑**：
```
第一层：Grep 快速召回（每波次执行）
  1. 从 details-lock.md 提取每个锁定值的"搜索关键词列表"
  2. 用 Grep 跨全部已写章节扫描
  3. 找到所有出现位置后，检查：
     - 同一锁定值的关键词是否一致（"右手小指"不应和"左手小指"共存）
     - 时序性锁定值是否在正确章节范围内（"念园"不应出现在ch27之前）
     - 数字是否一致（不应同时出现"12%"和"7%"指代同一事物）
  4. 输出冲突报告

第二层：LLM 归一化比对（仅终验时执行）[v2-changed]
  - 对 Grep 未能覆盖的变体表述（"那只残缺的手""握不紧的拳"等），
    由终验 Agent 在顺序通读时识别
  - 不在每波次执行（成本太高）
  - details-lock 的"搜索关键词列表"持续维护，减少对 LLM 归一化的依赖
```

### 方案 E：状态连续性扫描

**对应根因**：R5

**改动位置**：波次后处理步骤 + Phase 7 终验

**扫描逻辑**：
```
相邻章检查（每波次执行）：
  FOR i = 1 to current_chapter - 1:
    ch_i_end = event-ledger 第 i 章结束状态
    ch_i1_start = event-ledger 第 i+1 章开始状态（或从正文开头提取）

    检查6个维度：
    1. 地点连续性
    2. 人身自由连续性
    3. 伤情连续性
    4. 认知状态连续性
    5. 关系状态连续性
    6. 道具/资产连续性

    发现断裂 → 标记

全局状态约束检查（每波次执行）[v2-new]：
  除了相邻章检查，还需要跨多章扫描以下硬约束：
  - "被拘留"状态持续期间，该角色不应出现在任何自由活动场景
    → 扫描 event-ledger 中 人身自由="被拘留" 的记录
    → 在后续所有章节中 Grep 该角色名 + 检查场景描述
  - "某角色离开某地"后，不应在后续章节中无交代地出现在该地
  - "某道具被A拿走"后，不应在后续章节中出现在原位置

扫描范围规则 [v2.1-new]：
  - 短篇（≤50章）：全量 Grep 全部已写章节
  - 长篇（>50章）：增量扫描——只扫本波次新写章节 + 前后各5章
    + event-ledger 中状态变更涉及的历史章节
  - 终验阶段始终全量扫描
```

### 方案 F：审校强制触发 + 双层门禁

**对应根因**：R3

**改动位置**：writing-loop.md 波次后处理

**[v2-changed] 双层门禁设计**（采纳 Codex CLI "不要一刀切阻断"建议）：

```
门禁分两层：

Blocking Critical（硬阻断，必须修复才能继续下一波次）：
  - 人物生死状态矛盾（死人复活/活人消失）
  - 人身自由状态矛盾（被拘留的人出现在外面）
  - 地点硬冲突（上章逃离A地，下章无交代又出现在A地）
  - details-lock 中的锁定值被违反

Non-blocking Critical（记录+修复队列，不阻断主流水线）：
  - 风格轻微漂移
  - 过渡不够自然（但不影响逻辑）
  - 称呼偶尔不统一
  - 次要角色行为轻微偏移

Warning / Suggestion：
  - 记录到 review-notes.md
  - 全书完成后统一处理

防频繁停线机制 [v2-new]：
  - 同一波次内 Blocking Critical 修复最多2轮
  - 2轮修复后仍有 Blocking Critical → 自动降级为串行模式完成剩余章节
  - 降级后的串行模式可以更精细地控制质量
```

**审校触发规则**（不可跳过）：
```
每波次完成后，无条件触发审校：
  - 范围 = 本波次全部章节 + 前后各1章
  - 检查项（在原有6项基础上新增）：
    [新增] details-lock 一致性（方案D结果）
    [新增] event-ledger 状态连续性（方案E结果）
    [新增] 场景转换自然度
    [新增] 角色身份阶段合规性（检查 details-lock 的身份阶段表）
```

## 四、优先级冲突规则

**[v2-changed]** 采纳 Codex CLI "event-ledger 不应高于前章正文"建议：

```
当信息源冲突时，按此优先级裁决：

details-lock.md（事实锁定表）
  > 已修复章节正文（canonical，即经过审校确认的正文）
    > event-ledger.md（由正文投影生成，是正文的派生物）
      > 未修复章节正文
        > 原作内容
          > Agent 自行编造（最低优先级）

即：
1. 锁定表是最高权威——所有细节以锁定表为准
2. 正文高于 ledger——当正文和 ledger 冲突时，优先修正 ledger（因为 ledger 是从正文提取的）
3. 已修复的正文 > 未修复的正文
4. Agent 绝不可自行编造锁定表中未列的具体数字/物理细节
```
