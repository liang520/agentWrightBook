### [Important] 遗漏的第六条根因

对照 logic-review-report.md 的实际问题，有一类问题不在 R1-R5 的覆盖范围内：

**R6：配置阶段的角色/设定规划粒度不足，导致写作时 Agent 需要"现编"细节。**

典型案例：
- **C04**（"季飞"身份混乱）：character-map 中季飞的角色定位不够清晰（是独立角色还是季野的代号），导致 ch12 的 Agent 提前使用了"季飞"称呼。
- **W05**（季阳定位模糊）：角色映射表写了"暗中投靠季野"但没有分配具体场景，结果全书没有落实这个设定。
- **W06**（季飞身份与"第五个儿子"矛盾）：同一个角色在映射表中有两层含义但未做区分。
- **C06/C07**（两封信、东港保险柜来历不明）：关键道具的来龙去脉在配置阶段没有规划清楚。

这类问题的根因不是"并行 Agent 无统一事实源"（R1），而是**事实源本身不够完整**。details-lock.md 能锁住已知事实，但不能补足未规划的事实。

**建议**：在 Phase 3 配置阶段增加一个"关键道具/身份链条审查"步骤。对 character-map 和 setting-map 中的每个核心角色/道具，要求明确以下字段：
- 身份链条（谁在什么时候叫什么名字、为什么改名）
- 关键道具归属链（道具在哪里、谁拿着、何时转手）
- 设定完整性检查（映射表左列的每个设定，右列是否有足够细节支撑全书使用）

这可以作为**方案 A 的前置增强**，不需要独立成为新方案。

## 三、方案间的依赖关系与执行顺序

### 依赖图

```
方案 A (details-lock) ←── 方案 D (事实一致性扫描，读取 A 的数据)
                      ←── 方案 C (波次并行，Agent prompt 包含 A)
                      ←── 方案 G (终验消费 A 的数据)

方案 B (event-ledger) ←── 方案 E (状态连续性扫描，读取 B 的数据)
                      ←── 方案 C (波次并行，Agent 每章输出 delta 写入 B)
                      ←── 方案 G (终验消费 B 的数据)

方案 C (波次并行)     ←── 方案 F (每波次后触发审校，依赖 C 的波次边界)

方案 D (事实扫描)     ←── 方案 F (审校包含 D 的结果)
方案 E (状态扫描)     ←── 方案 F (审校包含 E 的结果)

方案 F (审校门禁)     → 独立执行，但依赖 D/E 的输出
方案 G (终验通读)     → 末端执行，消费 A/B/F 的产出物
```

### 结论

**无循环依赖**。依赖关系是单向的：A/B 是数据基础，C 是执行框架，D/E 是检测引擎，F 是门禁，G 是终验。

**执行顺序建议（实施改动的先后）**：

1. **第一批（数据基础）**：方案 A + 方案 B + 章节输出契约（第三节）+ 优先级冲突规则（第四节）
2. **第二批（执行框架）**：方案 C + 方案 F
3. **第三批（检测引擎）**：方案 D + 方案 E
4. **第四批（终验增强）**：方案 G

方案文档中的文件变更清单（第五节）列出了所有需要改动的文件，但**没有标注实施批次**。建议补充。

## 四、落地可行性分析

### 4.1 details-lock.md 在并行写作中的并发安全

**问题**：方案 A 规定"Agent 不得编造表中未列的具体数字/物理细节；如需新增，必须先追加到此表"。但在并行模式下，多个 Agent 同时写作，如果两个 Agent 都需要新增事实，会发生写冲突。

**当前方案的处理**：方案 C 的"串行主干期"（前4章串行）确实能在并行开始前建立核心事实基线。但问题出在波次并行期——如果 ch15 和 ch17 并行写作，两个 Agent 都发现需要新增一个地名，怎么办？

**建议**：
- 方案中已有"Agent 不得自行编造"的禁令，但需要明确**替代方案**：Agent 发现需要新事实时，应在章节正文中使用**模糊措辞**（如"那栋楼"而不是"47层大楼"），并在 chapter_delta 的 `new_facts` 字段中标记为"待锁定"。
- 每波次完成后的后处理步骤中，增加一步**"锁定待定事实"**：汇总本波次所有 Agent 的 `new_facts`，统一裁决后写入 details-lock.md，然后回填章节中的模糊措辞。
- 这样 details-lock.md 永远只在串行阶段（主干期或波次间隙）被修改，不存在并发写入问题。

### 4.2 event-ledger.md 的并发追加

**问题**：方案 B 规定"每章写完后，Agent 必须输出一条 chapter_delta，追加到此表"。并行模式下多个 Agent 同时追加，如果用文件追加（append），多个进程同时写同一个文件会导致内容交错。

**当前方案的处理**：方案 B 已写明"并行模式下，每批次完成后统一回填"。这是正确的。

**但需要明确执行机制**：
- Agent 的 chapter_delta 应该**输出到各自的章节目录或返回值中**，而不是直接追加到 event-ledger.md。
- 波次后处理时，由**单一的协调进程**按章号顺序将各 Agent 的 delta 合并写入 event-ledger.md。
- 建议在方案 B 中明确这个两段式流程（Agent 输出 delta -> 协调进程合并写入），避免实施时的歧义。

### 4.3 chapter_delta JSON 对 prompt 长度的影响

**问题**：每个写作 Agent 的 prompt 需要包含 details-lock.md + event-ledger.md + 原有的全套 config/context。随着写作推进，event-ledger.md 会线性增长（32 章约 32 行，每行 100-200 字，总计 3200-6400 字），details-lock.md 也会增长。

**评估**：
- 当前 Agent prompt 已包含：style.md（约 1000 字）+ memory-outline.md（约 2000 字）+ recent-context.md（变长）+ character-map.md（约 2000 字）+ setting-map.md（约 1500 字）+ compression-map.md（约 2000 字）+ foreshadowing.md（约 1000 字）+ 原作章节（约 2000 字）+ 前章注入（约 1500 字）。总计约 14000 字。
- 新增 details-lock.md（预估 1000-2000 字）+ event-ledger.md（预估 3000-6000 字）+ chapter_delta 格式说明（约 500 字）。
- **总增量约 4500-8500 字**，即 prompt 增长 30%-60%。

**结论**：对于当前模型的 context window（100K-200K tokens），这个增量完全可以承受。但对于短篇（32 章），event-ledger 全量加载是合理的；**如果将来用于长篇（200+ 章），event-ledger 需要分段加载**（只加载最近 20 章 + 与当前章有强耦合关系的历史章节）。

**建议**：在方案 B 中增加一条规则："当 event-ledger 超过 50 条时，写作 Agent 只加载最近 20 条 + 涉及当前章核心角色的历史条目。终验和状态扫描仍读取全量。"

### 6.1 实际写作中出现但方案未覆盖的问题类型

对照 logic-review-report.md 的完整问题清单，以下问题类型**不在 7 个方案的覆盖范围内**：

| 问题 | 类型 | 方案是否覆盖 | 说明 |
|------|------|-------------|------|
| C04（季飞身份提前使用） | 角色身份链混乱 | **未覆盖** | details-lock 锁定的是物理细节，不涵盖"角色在什么阶段叫什么名字"这种时序性身份规则 |
| C06（两封"念念亲启"的信） | 道具重复 | **部分覆盖** | details-lock 可以锁定"信的数量和来源"，但前提是配置阶段能预见到需要锁定 |
| C07（东港保险柜来历不明） | 因果链缺失 | **未覆盖** | 这是情节设计问题，不是并行写作问题。event-ledger 只记录已写的事件，不能补足缺失的铺垫 |
| W05（季阳设定未落实） | 角色弧未闭合 | **未覆盖** | details-lock 锁定事实但不追踪"设定是否被使用"。这更接近伏笔追踪的范畴 |
| W10（季澄突然拥有大量信息） | 信息来源缺铺垫 | **未覆盖** | 同 C07，属于情节设计问题 |

**建议增加的覆盖项**：

1. **角色身份时间线**：在 details-lock.md 中增加一个"身份阶段表"区块：
   ```
   ## 角色身份阶段
   | 角色 | 阶段 | 使用名称 | 起始章 | 结束章 |
   |------|------|---------|--------|--------|
   | 季野 | 被洗脑前 | 季野/野哥 | 01 | 13 |
   | 季野 | 伪装季飞 | 季飞/五少 | 14 | 17 |

   这样 ch12 的 Agent 就不会提前使用"季飞"。

2. **关键道具追踪**：在 event-ledger.md 中增加一个"道具/信息转移"字段，或新建 `details-lock.md` 的"道具归属表"区块：
   ```
   ## 关键道具归属
   | 道具 | 当前持有者 | 当前位置 | 首次出现章 | 转移记录 |

   | 老爷子的信 | 沈念 | 随身 | 新18 | 18章车上打开 |

3. **角色设定落实追踪**：在 foreshadowing.md 中扩展，不仅追踪伏笔，还追踪"角色设定承诺"。例如 character-map 中写了"季阳暗中投靠季野"，就应该在 foreshadowing 中有一条对应条目，确保这个设定在全书中有至少一个场景落实。

### 6.2 Codex CLI 提出的"事实变更治理"是否被充分纳入

方案 A 的"Codex CLI 补强建议落地"部分提到了三点：
1. 表头增加 version 字段 -- **已纳入**
2. 每次修改追加变更日志 -- **已纳入**
3. 修改后自动定位受影响的已写章节（通过 Grep 关键词）-- **已纳入但不够具体**

**问题**：第3点"自动定位受影响章节"在方案中只提了一句，但没有回答关键问题：**定位到之后怎么办？**

可能的场景：details-lock 中"断指位置"从"左手"改为"右手"（修复 W01），Grep 找到 ch11 中出现了"左手小指"。然后呢？
- 是自动替换？（简单替换可能破坏上下文）
- 是标记为"待修复"等下次审校时处理？（可能忘记）
- 是立即暂停写作、修复后继续？（影响效率）

**建议**：明确变更治理的完整流程：
```
details-lock 变更 → Grep 定位受影响章节 → 分级处理：
  - 简单替换型（如"左手"→"右手"）：自动替换 + 记录到变更日志
  - 语义变更型（如改变断指原因）：标记为 Critical，阻断当前波次，修复后才能继续
  - 未来章节影响型（如改变了尚未写到的伏笔回收方式）：更新 foreshadowing.md + compression-map.md
```

## 七、方案 D（事实一致性扫描）的技术局限

方案 D 的扫描逻辑是：从 details-lock 提取关键词 -> Grep 搜索 -> 规则比对。

**Codex CLI 补强建议**提到了"归一化比对"（同义词、代词识别），但方案中没有给出具体实现方案。

**实际难度**：
- "季野的断指"可以表述为"右手小指缺了半截""那只残缺的手""他攥了攥右手，小指的空缺让拳头永远握不紧""断了半截的手指"等等。
- Grep 只能匹配字面关键词，无法覆盖这些变体。
- 如果要用 LLM 做归一化识别，每次扫描的成本会大幅上升。

**建议**：
- 方案 D 的 Grep 层作为**快速召回**（高召回率、低精确率）是合理的。
- "归一化比对"应该**只在终验（方案 G）中执行**，而不是每波次都做。每波次只做 Grep 层的字面匹配，成本可控。
- 在 details-lock.md 中，为每个锁定值维护一个**"别名/变体"列表**，供 Grep 使用：
  ```
  | 角色 | 细节项 | 锁定值 | 搜索关键词列表 |
  |------|--------|--------|--------------|
  | 季野 | 断指位置 | 右手小指缺半截 | 右手小指,小指缺,断指,残缺的手,握不紧的拳 |
  ```
  这样 Grep 的召回率可以显著提升，而不需要每次都调用 LLM。

## 八、具体修改建议汇总

### [Critical] 必须修改

1. **方案 A 增加"角色身份阶段表"和"关键道具归属表"**。否则 C04、C06、C07 类问题仍会复现。

2. **方案 B 明确并发安全机制**。当前"并行模式下每批次完成后统一回填"的表述正确但不够具体，需要明确 delta 的暂存位置（建议存在各章节的 JSON sidecar 文件中，如 `chapters/020.delta.json`），以及由谁（协调进程）在什么时机（波次后处理第 0 步）合并写入 event-ledger.md。

3. **补充 details-lock 变更后的处理流程**（定位受影响章节后的分级处理规则），否则"事实变更治理"只完成了一半。
