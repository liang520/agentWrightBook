# `/novel` 统一指令使用说明

**核心交互原则：所有环节都是选择题，回复数字即可。**

---

## 子命令速查表

| 命令 | 功能 | 典型场景 |
|------|------|---------|
| `/novel` | 自动检测状态跳转 | 每次新会话的入口 |
| `/novel status` | 查看所有项目进度 | 了解当前全局状态 |
| `/novel analyze` | 分析原作结构 | 导入新原作后 |
| `/novel new` | 配置新书 | 原作分析完毕后 |
| `/novel write` | 开始/继续写作 | 进入持续写作模式 |
| `/novel continue` | 同 write | 恢复中断的写作 |
| `/novel review [范围]` | 审校章节 | 手动检查质量 |
| `/novel compress` | 手动记忆压缩 | 通常不需要（写作时自动执行） |
| `/novel verify` | 全书 10 维度校验 | 全书写完后 |
| `/novel fix` | 自动修复问题 | 校验发现问题后 |
| `/novel auto` | 全自动流水线 | 从零开始一键完成整本书 |

---

## 典型工作流

### 全新书（推荐）

```
/novel auto
→ 选原作 → 选题材 → 自动分析
→ 选世界观 → 选风格 → 选书名 → 角色/设定映射
→ 持续自动写作（每5章审校、每10章压缩+汇报）
→ 全书校验（10维度）→ 自动修复
```

### 续写（新会话恢复中断的书）

```
/novel
→ 自动检测到 status="writing" → 加载三层记忆 → 从断点继续
```

### 仅校验已完成的书

```
/novel verify
→ 选小说 → 10维度检查 → 生成报告
→ 如有问题：/novel fix
```

### 手动审校特定范围

```
/novel review 20-30
→ 检查第20-30章 → 展示结果 → 选择修复方式
```

---

## 写作阶段行为

- **持续自动**：写到 context 用完或用户叫停，不会主动暂停
- **每 5 章**：启动后台审校 Agent，检查映射泄漏和一致性
- **每 10 章**：记忆压缩 + 检查点汇报（汇报后继续，不等待）
- **新实体**：自动生成映射，在检查点汇报中列出
- **中断安全**：meta.json 每章更新，任意时刻中断都可恢复

---

## 常见问题

**Q: 以前的 `/write-chapter`、`/batch-write` 等指令还能用吗？**
A: 旧指令已归档到 `.claude/skills-archive/`。所有功能已整合到 `/novel` 的子命令中。

**Q: 写作时怎么叫停？**
A: 直接发送任意消息即可中断。meta.json 已记录最后完成的章节，下次 `/novel` 自动恢复。

**Q: 记忆压缩是自动的吗？**
A: 是。写作每 10 章自动压缩。也可手动 `/novel compress`。

**Q: 校验有哪 10 个维度？**
A: 映射泄漏扫描、章节完整性、字数分布、等级体系一致性、角色名一致性、伏笔闭合、风格一致性、记忆档案完整性、审校发现汇总、非关键建议。

**Q: 如何同时管理多本小说？**
A: `/novel status` 查看所有项目。选择具体小说时以选择题呈现。

# `/short` 短篇改编指令使用说明

**核心原则：全自动无人参与，从分析到成稿一键完成。**

---

## 子命令速查表

| 命令 | 功能 | 典型场景 |
|------|------|---------|
| `/short` | 自动检测状态跳转 | 每次新会话的入口 |
| `/short status` | 查看短篇项目进度 | 了解当前全局状态 |
| `/short analyze` | 分析原作（含压缩标注） | 导入新原作后 |
| `/short plan` | 骨架提取 + 压缩规划 | 分析完毕后 |
| `/short new` | 自动配置新书 | 压缩规划完成后 |
| `/short write` | 开始/继续写作 | 进入持续写作模式 |
| `/short continue` | 同 write | 恢复中断的写作 |
| `/short review [范围]` | 审校章节 | 手动检查质量 |
| `/short compress` | 手动记忆压缩 | 通常不需要（写作时自动执行） |
| `/short verify` | 全书 11 维度校验 | 全书写完后 |
| `/short fix` | 自动修复问题 | 校验发现问题后 |
| `/short auto` | 全自动流水线 | 从零开始一键完成整本书 |

---

## 与 `/novel` 的区别

| 维度 | `/novel`（长篇仿写） | `/short`（短篇改编） |
|------|---------------------|---------------------|
| 目标字数 | 与原作等长 | 4.5万-5万字 |
| 章节映射 | 1:1 | 多对一（压缩映射表驱动） |
| 用户参与 | 配置阶段有选择题 | 全程无人参与 |
| 新增阶段 | 无 | 骨架提取 + 压缩规划 + 独立校验 Agent |
| 校验维度 | 10 维度 | 11 维度（+压缩质量总评） |

---

## 典型工作流

### 全自动（推荐）

```
/short auto
→ 逐章分析原作（含压缩标注）
→ 骨架提取 + 压缩映射表 + 独立 Agent 校验
→ 自动配置（世界观/风格/角色/设定映射）
→ 按映射表逐章改写（每5章审校、每10章压缩+汇报）
→ 全书校验（11维度）→ 自动修复
```

### 续写（新会话恢复中断的书）

```
/short
→ 自动检测到 mode="short" + status="writing"
→ 加载三层记忆 + 压缩映射表 → 从断点继续
```

---

## 压缩规划的独立校验

Phase 2 生成压缩映射表后，会启动独立校验 Agent 检查 6 项：
1. 字数达标（≥ 45000）
2. 原作章节全覆盖
3. 伏笔链完整
4. 节奏合理
5. 不可压缩章节全保留
6. 因果链完整

校验不通过则自动修正，最多 3 轮。

---

## 常见问题

**Q: 压缩后的小说会丢失重要情节吗？**
A: 不会。压缩的是描写、过渡、灌水内容，情节因果链和角色关系完整保留。节奏会更快，但逻辑不变。

**Q: 目标字数是怎么计算的？**
A: 取原作字数的 50% 和 45000 的较大值，但不超过 55000。

**Q: 短篇改编也有三层记忆吗？**
A: 有，与 `/novel` 完全一致。每10章压缩归档，保证一致性。

**Q: 校验比 `/novel` 多了什么？**
A: 多了第 11 维度"压缩质量总评"：检查总字数达标、与映射表的偏差、因果链完整性、节奏曲线。

# `/condense` 精炼指令使用说明

**核心原则：对已完成小说进行精炼压缩，保持故事内容不变，削减冗余文字。**

---

## 子命令速查表

| 命令 | 功能 | 典型场景 |
|------|------|---------|
| `/condense` | 自动检测状态跳转 | 每次新会话的入口 |
| `/condense status` | 查看精炼项目进度 | 了解当前全局状态 |
| `/condense diagnose` | 诊断原稿 | 首次精炼时 |
| `/condense write` | 开始/继续精炼写作 | 诊断完成后 |
| `/condense continue` | 同 write | 恢复中断的精炼 |
| `/condense verify` | 终验（5维度） | 精炼写完后 |
| `/condense fix` | 自动修复问题 | 终验发现问题后 |
| `/condense auto` | 全自动流水线 | 一键完成精炼 |

---

## 与 `/novel` 和 `/short` 的区别

| 维度 | `/novel`（长篇仿写） | `/short`（短篇改编） | `/condense`（精炼） |
|------|---------------------|---------------------|-------------------|
| 输入 | 原作（sources/） | 原作（sources/） | 已完成小说（novels/） |
| 目标 | 等长仿写 | 压缩改编（4.5-5万字） | 精炼到目标字数 |
| 是否改编 | 是（换角色/设定） | 是（换角色/设定+压缩） | 否（原文精炼） |
| 章节结构 | 1:1 | 多对一（压缩映射） | 1:1（保持不变） |
| 存储 | novels/{name}/ | novels/{name}/ | novels/{name}-condensed/ |
| 原书 | 只读 sources/ | 只读 sources/ | 严格只读 novels/{name}/ |
| 校验维度 | 10 | 11 | 5 |
| 用户参与 | 配置有选择题 | 全程无人 | 全程无人 |

---

## 典型工作流

### 全自动（推荐）

```
/condense auto
→ 选择已完成的小说 → 精确统计 + 逐章分级
→ 生成精炼计划（condense-plan.md）
→ 逐章精炼重写（每5章预算检查、每10章记忆压缩+汇报）
→ 终验（5维度）→ 自动修复
```

### 续写（新会话恢复中断）

```
/condense
→ 自动检测到 mode="condense" + status="writing"
→ 加载记忆 + 精炼计划 → 从断点继续
```

### 仅诊断（看看能压缩多少）

```
/condense diagnose
→ 选小说 → 逐章分级 → 输出精炼计划（不开始写作）
```

---

## 精炼方法论

按 A/B/C/D 四级分级，不同等级执行不同力度的精炼：

| 等级 | 力度 | 主要手法 |
|------|------|---------|
| A 高度可精炼 | 砍 25%+ | 环境描写铺排→1句意象、删战斗流水账、心理独白→点睛 |
| B 中度可精炼 | 砍 10-25% | 压缩但保留氛围、删废话对话、缩短过渡 |
| C 轻度可精炼 | 砍 5-10% | 仅措辞精炼（拆长句、删冗余形容词） |
| D 不可精炼 | 0% | 直接复制原章 |

**绝不动**：事件清单要点、伏笔句、章尾钩子、角色口头禅、情绪高/低点。

---

## 常见问题

**Q: 精炼会改变故事内容吗？**
A: 不会。角色名、设定名、情节走向、因果链全部保持原样。只精炼文字表达。

**Q: 原书文件会被修改吗？**
A: 绝不会。原书目录严格只读，精炼稿存在独立的 `{name}-condensed/` 目录。

**Q: 精炼后的记忆系统怎么处理？**
A: memory-outline 和 archives 从原书复制（故事不变），recent-context 从空开始逐章重建（文字变了）。

**Q: 字数统计口径是什么？**
A: 去空格、去 Markdown 标记后的字符数（含标点），详见 CLAUDE.md "字数统计规范"。

**Q: 终验只有 5 个维度？**
A: 是。不需要映射泄漏检查（没有映射）、不需要压缩质量检查（不是改编）。5 维度：总字数、伏笔完整性、情绪曲线、A/B对照、因果链。

## 压缩方法论

三种压缩策略，按章节映射关系选用：

| 策略 | 适用场景 | 主要手法 |
|------|---------|---------|
| 保留/精炼 | 单章→单章，内容密度高 | 同 `/condense` 的 B/C 级精炼 |
| 合并 | 多章→单章，功能相近 | 提取核心事件编织成完整章节，删重复过渡 |
| 浓缩 | 单章大幅压缩 | 同 `/condense` 的 A 级手法 |

另外可直接删除纯支线/灌水章节（主线无因果依赖）。

**绝不动**：事件清单要点、伏笔句、章尾钩子、角色口头禅、情绪高/低点。

---

## 常见问题

**Q: `/compress` 和 `/condense` 有什么区别？**
A: `/condense` 只做文字精炼（不合并章节、不删支线），最多压缩 ~25%。`/compress` 可以合并章节和删支线，压缩 ~50%。两者都不改变角色名和设定名。

**Q: `/compress` 和 `/short` 有什么区别？**
A: `/short` 会换角色名和设定名（改编），`/compress` 保持原样（不改编）。两者都支持合并章节和删支线，压缩能力相当。

**Q: 原书文件会被修改吗？**
A: 绝不会。原书目录严格只读，压缩稿存在独立的 `{name}-compressed/` 目录。

**Q: 伏笔会丢失吗？**
A: 不会。被删除章节的伏笔会按方向性约束迁移到相邻保留章（埋设向前、回收向后）。

**Q: 字数统计口径是什么？**
A: 去空格、去 Markdown 标记后的字符数（含标点），详见 CLAUDE.md "字数统计规范"。

**Q: 终验有哪 8 个维度？**
A: 总字数、章节完整性、伏笔完整性、因果链、情绪曲线、原作覆盖率、风格一致性、A/B对照。
